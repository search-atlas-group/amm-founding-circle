# Founding Circle Dashboard

One local page, six tabs -- Connection Sentinel, Bug Hunter, Content QA,
Lead Grader, Penny Dashboard, Outbound Engine -- each showing that tool's
own latest report. Click **Refresh** on any tab and it re-runs that tool's
own command (its normal `run.py` / CLI, nothing reimplemented) and pulls
the freshly generated report straight back into the tab, no page reload.

**Why this exists:** JD reviewed the six tools' raw HTML outputs and called
them not good enough for a client-facing report -- inconsistent, flat,
undifferentiated documents rather than something you'd present live. This
tool does two things: (1) it's the reason all six tools now share one
consistent, professional design system instead of six near-identical but
subtly different ad hoc styles (see "The design system" below), and (2) it
wraps all six in one client-report-quality dashboard instead of six
separate files a member has to hunt down individually.

## How to run

```bash
cd tools/dashboard
python3 run.py                # opens http://127.0.0.1:58822 in your browser
python3 run.py --port 58900   # pick a different port
python3 run.py --no-browser   # start the server only
```

Zero pip installs -- this is stdlib `http.server` + `subprocess` only, same
dependency-light bar as every other tool in this repo. Stop it with
`Ctrl+C`.

The first time you open a tab, it shows whatever report that tool has
already generated (its demo/example data, out of the box). Click
**Refresh** to run it for real against your own config.

## What "Refresh" actually does

Each tab is wired to that tool's own, documented command:

| Tab | Command it runs | Report it reads back |
|---|---|---|
| Connection Sentinel | `python3 sentinel.py --config connections.yaml --once` | `status.html` |
| Bug Hunter | `python3 run.py --no-deliver` | newest `reports/bug-hunter-*.html` |
| Content QA | `python3 run.py examples/sample-draft.md --client acme-example --no-llm` | `reports/acme-example-sample-draft.html` |
| Lead Grader | `import --from-file ...` then `digest` | newest `output/digest-*.html` |
| Penny Dashboard | `python3 run.py generate` | `out/owner.html` **or** `out/clients/<client>.html` (toggle) |
| Outbound Engine | `pipeline --dry-run` then `report` | `reports/weekly-report.html` |

Nothing here reimplements a tool's crawling / grading / margin-math logic
-- the dashboard only shells out to the command already documented in that
tool's own README, then reads the HTML file that command already writes.
Swap the demo commands above for your own real client args (edit
`dashboard/tools_config.py`'s `run_steps`) once you're past the demo.

A non-zero exit code from a step is **not** treated as a failure by
itself -- Bug Hunter and Connection Sentinel both deliberately exit
non-zero to signal findings (a critical issue found, a connection down),
which is the tool doing its job. The dashboard only reports a real failure
when no report file shows up at all.

## The design system

All six tools' own HTML renderers were re-skinned to share one design
system -- `theme.py`, vendored into each tool's own package so every tool
still runs completely standalone (a member cloning just `tools/bug-hunter`
never needs this `dashboard/` folder to exist). The canonical copy lives at
`dashboard/theme.py`; the six vendored copies are byte-identical unless a
tool needed a small local addition (see each tool's own render module).

The look is adapted from this repo's own `skills/html-reports/atlas`
archetype -- the "leadership BI dashboard" shape (hairline cards, tabular
numerals, letterspaced uppercase labels, a KPI strip, status pills) that
already matches the house style: light mode, one muted accent color, no
gradients, no neon. Reusing an already-vetted in-repo system, rather than
inventing a seventh look, is what makes six different tools read as one
product.

## Penny Dashboard's owner/client toggle

Penny Dashboard's tab has two sub-views (buttons above the report):
**Owner view (internal)** and **Client-safe view**. Switching between them
only reads a different already-rendered file -- it never re-runs anything.
The client-safe file is generated by `penny_dashboard.render.render_client_safe_view`,
which can only ever receive a `ClientSafeView` object with no margin/cost
attribute on it at all (see `visibility.py`) -- the re-skin here only
touched markup and CSS, never that data contract. Never remove that
guarantee to make the dashboard "look" more unified; margin/cost data must
stay structurally absent from anything a client could see.

## Files

```
dashboard/
  run.py            CLI entrypoint -- python3 run.py
  dashboard/
    server.py        stdlib HTTP server: the tab shell + /api/tab + /api/run
    shell.py          the tab-bar + iframe-pane page + refresh JS
    tools_config.py   per-tool adapter: which command to run, which file to read
    theme.py          the shared design system (also vendored into each tool)
```

## Common mistakes

- **Running a real client's Lead Grader digest expecting numbers with no
  LLM key configured.** Grading fails closed on purpose (per Lead Grader's
  own design) rather than guessing a grade -- the tab will show whatever
  the last real grading run produced, or "nothing graded yet" if there
  hasn't been one. Set `ANTHROPIC_API_KEY` / `OPENROUTER_API_KEY` in
  `tools/lead-grader/.env` for a real digest.
- **Assuming a red "Refresh failed" banner means the tool is broken.** Read
  the log underneath it first -- for Bug Hunter/Connection Sentinel a
  non-zero exit is usually the tool correctly reporting a finding, not a
  crash (see "What Refresh actually does" above); the dashboard already
  accounts for this and only flags it red when no report file appears.
- **Editing the demo `run_steps` in place and forgetting they're shared.**
  `tools_config.py`'s commands are meant to be edited once you have real
  client config -- they aren't reset by anything, so a stale hard-coded
  client slug will silently keep showing that one client's data.

`SKILL.md` in this folder wraps the tool so an agentic CLI (Claude Code,
